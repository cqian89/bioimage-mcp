# Phase 5.1: Research Dynamic Discovery Standardization - Research

**Researched:** 2026-01-25
**Domain:** Dynamic Function Discovery & Introspection
**Confidence:** HIGH

## Summary

This research investigates the current state of dynamic function discovery across the `bioimage-mcp` ecosystem. Currently, discovery is split between "In-Process" adapters (server imports library) and "Out-of-Process" protocol calls (`meta.list` and `meta.describe` via subprocess). The implementations vary significantly across tool packs, with `trackpy` using `numpydoc` for rich extraction, while `cellpose` and `base` use simpler `inspect.signature` based approaches.

Standardization is required to support requirement **GEN-01** (dynamic discovery for scipy/numpydoc) and to reduce boilerplate in new tool packs.

**Primary recommendation:** Standardize on a `numpydoc`-based introspection protocol, provided via a shared utility in `bioimage_mcp.runtimes.introspect`, which tool packs can import to implement `meta.list` and `meta.describe` with minimal code.

## Architecture Patterns

### Recommended Protocol Implementation
Tool packs should implement a standard "Dynamic Adapter" that handles `meta.list` and `meta.describe` by calling shared server-provided utilities.

### Pattern 1: Subprocess Enrichment
The core server loads a `manifest.yaml` with `dynamic_sources`. It then calls the tool's entrypoint with `fn_id: meta.list` to discover functions it doesn't already know about.

### Pattern 2: Schema Enrichment (Lazy Load)
The core server calls `meta.describe` when a function is first "described" or "run" to populate the full `params_schema`. This results are cached in the server's `SchemaCache`.

### Anti-Patterns to Avoid
- **Hardcoded lists:** Avoid maintaining `functions` lists in `entrypoint.py` (Cellpose pattern). Use module scanning.
- **Manual description copies:** Don't copy docstrings into `descriptions.py`. Let `numpydoc` extract them.
- **In-Process Reliance:** Never assume the server can import tool dependencies. Always provide a subprocess fallback.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Docstring Parsing | Regex parsers | `numpydoc` | Scientific Python docstrings (Numpy format) are complex and standard. |
| Type Inference | Custom `isinstance` checks | `inspect.signature` | Handles defaults, type hints, and kind correctly. |
| Port Identification | Hardcoded lists | `bioimage_mcp` model matching | Use `IOPattern` definitions and artifact type detection. |

## Common Pitfalls

### Pitfall 1: Environment Mismatch
**What goes wrong:** `numpydoc` or the library itself is missing in the server environment.
**Why it happens:** Server environment is kept "light" while tool packs are "heavy".
**How to avoid:** Always run `meta.*` commands *inside* the tool's conda environment via `conda run`.

### Pitfall 2: I/O Pattern Confusion
**What goes wrong:** Discovery identifies an image input (e.g., `image`) as a scalar parameter.
**Why it happens:** Introspection sees a parameter but doesn't know it's an "Artifact Port".
**How to avoid:** Standardize `ARTIFACT_INPUT_PARAMS` (like `trackpy` does) or use type-hint matching (e.g., `np.ndarray` or `BioImageRef`).

### Pitfall 3: Version Drift
**What goes wrong:** Manifest says one thing, runtime library (e.g., cellpose) says another.
**Why it happens:** Library update in conda env doesn't trigger manifest update.
**How to avoid:** `meta.describe` must report `tool_version`. Server should invalidate schema cache when `tool_version` changes.

## Open Questions

1. **Scipy/Skimage Naming Conventions**: Do they provide enough consistency for automated I/O pattern mapping (e.g., everything in `skimage.filters` is `IMAGE_TO_IMAGE`)?
   - *Recommendation*: Use a module-to-pattern mapping configuration in the manifest.

2. **Dependency on `numpydoc`**: Should we bundle `numpydoc` in the base tool environment?
   - *Recommendation*: Yes, `bioimage-mcp-base` should include it.
