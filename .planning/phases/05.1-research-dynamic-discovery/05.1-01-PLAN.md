  <files>specs/024-meta-discovery-standardization/spec.md</files>
  <action>
Create `specs/024-meta-discovery-standardization/spec.md` documenting:

- Scope: tool-pack discovery protocol between server and tool envs.
- `meta.describe` request/response contract (reference existing spec, but restate the required fields):
  - Request: `{fn_id: "meta.describe", params: {target_fn: string}}`
  - Success: `ok: true`, `result.params_schema` (JSON Schema object), `result.tool_version` (string), `result.introspection_source` (string)
  - Error: `ok: false`, `error` is a *string* (not a dict)
  - Notes: server may receive either `{"result": ...}` or worker-shaped `{"outputs": {"result": ...}}`.

- `meta.list` request/response contract:
  - Request: `{fn_id: "meta.list", params: {}, inputs: {}}`.
  - Success: `ok: true`, `result.functions` list.
  - Minimal function entry required keys: `fn_id`, `name`, `summary`.
  - Recommended keys (optional): `module`, `io_pattern` (must match `IOPattern` enum values), `description`, `parameters`, `returns`, `tags`, `hints`.
  - Recommended `result.tool_version` + `result.introspection_source` (optional, but preferred).

- Naming and prefixing rules:
  - Tool packs SHOULD return library-level fn_ids (e.g. `scipy.ndimage.gaussian_filter`) and let the server apply env prefix when needed.
  - Tool packs MAY return already-prefixed ids; server must tolerate both.

- Version semantics:
  - `tool_version` is the underlying library version used to introspect (e.g. `cellpose.__version__`, `trackpy` version, `scipy` version).

Include at least one JSON example for each protocol and explicitly document backwards-compat expectations (server should be lenient; tool packs should be strict).
