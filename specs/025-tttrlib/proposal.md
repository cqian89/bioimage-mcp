**Spec ID**: 025-tttrlib  \
**Feature Branch**: `025-tttrlib`  \
**Status**: Draft  \
**Created**: 2026-01-14  \
**Target Version**: TBD

---

## 1. Executive Summary

This proposal adds a new tool pack built on `tttrlib` (https://docs.peulen.xyz/tttrlib/, https://github.com/Fluorescence-Tools/tttrlib) to enable **event-based** fluorescence workflows in bioimage-mcp:

1. **FCS / cross-correlation** via `tttrlib.Correlator` (multi-tau correlation).
2. **ICS** from scanning TTTR acquisitions via `tttrlib.CLSMImage` and its ICS routines.
3. **Single-molecule burst selection** via `TTTR.get_time_window_ranges` and intensity traces (time-window parameters follow tttrlib units; for `get_intensity_trace`, the implementation expects seconds).
4. **Photon-HDF5 import/export** for standardized exchange of photon-stream data.

Maintainer decisions applied:

- Use **Approach 2**: a **filtered adaptor exposing constructors/methods**, with **manual schemas generated by scraping tttrlib documentation**.
- Vendor a **minimal subset** of upstream `tttr-data` into `datasets/tttr-data/` with an informative `README.md`.
- Introduce a dedicated **`TTTRRef`** artifact type now.
- Include **Photon-HDF5 import/export** in v0.

Key integration constraints:

- `tttrlib` is a SWIG-wrapped C++ library with compiled dependencies; it must run in an **isolated tool environment**.
- The current bioimage-mcp core/base environment is Python 3.13; `tttrlib` is expected to be most reliable on **Python 3.12** (no guaranteed `py313` wheels/conda builds).
- Many `tttrlib` Python entry points use `*args/**kwargs` or dynamic getters, so signature-based introspection is not reliable; we will expose a curated API with **manual schemas**.

## 2. Motivation & Use Cases

bioimage-mcp currently supports pixel-based image processing (base toolkit), plotting (matplotlib), and phasor FLIM (phasorpy). Many fluorescence biophysics workflows begin one step earlier: **photon arrival events** in TTTR formats.

### 2.1 Primary user scenarios

| Scenario | Input | Outputs | Why tttrlib |
|---|---|---|---|
| FCS (auto / cross) | `TTTRRef` (`.spc`, `.ptu`, `.ht3`, Photon-HDF5) | correlation curve `TableRef` (+ optional plot) | Efficient multi-tau in C++ |
| ICS from scanning TTTR | `TTTRRef` with scan markers | ICS map `BioImageRef` + summary `TableRef` | Handles scan markers / binning |
| Burst selection (single molecule) | `TTTRRef` | burst ranges `TableRef`, traces `TableRef` | Fast time-window selection |
| Interop: share/standardize | `TTTRRef` (vendor formats) | `TTTRRef` (Photon-HDF5) | Open container for exchange |

### 2.2 Non-goals for the first milestone

- Full exposure of all SWIG-wrapped C++ APIs.
- GUI/interactive analysis.
- Streaming TTTR ingestion (batch files only).

## 3. Current State & Gap Analysis

### 3.1 Current state

- Tool packs are environment-isolated and exchange data via typed artifact references.
- Dynamic discovery exists for select libraries (skimage/xarray/pandas/matplotlib/phasorpy) and is filtered/curated via adapters and allowlists.
- Cellpose demonstrates class/object workflows via `ObjectRef`.

### 3.2 Gap

- No canonical artifact type for photon-stream files.
- No built-in ability to compute FCS/ICS or perform burst selection from TTTR event streams.
- No standardized import/export path for Photon-HDF5.

## 6. Filtered Exposure & Discovery Design (Approach 2)

We will implement **Approach 2**: a filtered adaptor exposing selected `tttrlib` constructors/methods.

### 6.1 Why an adaptor (vs wrapper-only)

`tttrlib` is large and domain-specific. A curated adaptor:

- exposes a consistent constructor/method workflow like other adaptors (notably Cellpose)
- allows selective expansion over time without redesigning the MCP surface
- provides a place to enforce I/O typing (`TTTRRef`, `TableRef`, `BioImageRef`) and safety rules

### 6.2 Manual schemas generated by scraping documentation

Because SWIG often obscures Python signatures (`*args/**kwargs`), we cannot rely on `inspect.signature` to generate accurate JSON schema.

Plan:

1. **Scrape tttrlib documentation** (source-of-truth) to extract parameter names, defaults, and usage patterns.
   - Sources include the published docs and the documentation sources in the `tttrlib` repo (e.g., `doc/modules/*.ipynb`, `.rst`, and Sphinx-gallery examples).
2. Generate a **versioned schema file** (YAML/JSON) committed into this repo, e.g. `tools/tttrlib/schema/tttrlib_api.json`.
   - Each entry defines: `fn_id`, summary/description, parameter JSON schema, and I/O typing.
   - The schema file must record the upstream `tttrlib` version it was scraped from.
3. The core-server `TttrlibAdapter` uses this schema file for `list`/`describe` without importing `tttrlib` in Python 3.13.
4. The tool runtime uses the same schema file to validate params and dispatch calls to real `tttrlib` functions.
5. Add a contract test that asserts the schema's recorded `tttrlib` version matches the tool environment lockfile/package version to prevent silent schema drift.

This preserves a stable MCP surface while still following the adaptor pattern.

### 6.3 Allowlisted surface (P0)

Initial allowlist mirrors tttrlib's actual class/method structure:

- `tttrlib.TTTR` — constructor, `.header`, `.get_time_window_ranges`, `.write`
- `tttrlib.Correlator` — multi-tau correlation
- `tttrlib.CLSMImage` — constructor, `.compute_ics`

## 7. Artifact / Ref Strategy

### 7.1 New: `TTTRRef` (introduced now)

Add a dedicated artifact type for photon-stream files.

- Type: `TTTRRef`
- Storage: file-backed (like `BioImageRef`)
- `format`: one of the supported container identifiers (e.g., `PTU`, `HT3`, `SPC-130`, `SPC-630_256`, `SPC-630_4096`, `HDF`)

Proposed minimal metadata fields (only what we can reliably extract at load time without fully parsing the file):

- `n_valid_events: int | None` (may require opening the file; populate lazily)
- `used_routing_channels: list[int] | None` (may require opening)
- `macro_time_resolution_s: float | None`
- `micro_time_resolution_s: float | None`

Note: The `format` field is set at load time based on file extension or explicit `container_type` parameter. Detailed metadata fields may be populated lazily when the file is accessed.

### 7.2 `TTTRRef` lifecycle

`TTTRRef` is file-backed: the raw photon-stream file (`.spc`, `.ptu`, `.ht3`, `.hdf5`) persists on disk. The reference remains valid as long as the underlying file exists and is accessible.

- **Input files**: Created via `tttrlib.TTTR(path, container_type)` constructor, which validates the path and returns a `TTTRRef` pointing to the original file location. Processing functions consume `TTTRRef` directly (loading internally as needed).
- **Exported files**: `tttrlib.TTTR.write` writes a new file (format inferred from extension) and returns a `TTTRRef` pointing to that location; output path handling follows the server write allowlist (inherits client permissions).

### 7.3 `ObjectRef` for derived/stateful objects

Use `ObjectRef` (existing type) only for derived or stateful objects that cannot be represented as simple file references:

- `tttrlib.CLSMImage` (reconstructed scanning image; may be exposed if ICS workflows need intermediate reuse)
- `tttrlib.Correlator` (if we expose stateful correlator objects)

Basic TTTR data does **not** require `ObjectRef`—use `TTTRRef` directly.

### 7.4 Downstream result artifacts

All processed/derived outputs use existing file-backed artifact types:

- `TableRef`: correlation curves, burst ranges, intensity traces (Parquet or CSV)
- `BioImageRef`: ICS maps, reconstructed images (OME-TIFF)
- `NativeOutputRef` (`json`): header export

These are persisted to the artifact store and remain valid beyond the session.

## 8. Proposed Public API Surface (v0)

### 8.1 Function table

The MCP function IDs mirror tttrlib's actual class/method structure:

| fn_id | input_mode | inputs | outputs | Notes |
|---|---|---|---|---|
| `tttrlib.TTTR` | `path` | (none) | `tttr: TTTRRef` | Constructor: validate path (read allowlist; inherits client permissions), set `format` and minimal metadata |
| `tttrlib.TTTR.header` | `artifact` | `tttr: TTTRRef` | `header: NativeOutputRef` (`json`) | Serialize `tttr.header.json` to an artifact |
| `tttrlib.TTTR.get_time_window_ranges` | `artifact` | `tttr: TTTRRef` | `ranges: TableRef` | Burst selection; mirrors `examples/single_molecule/plot_single_molecule_burst_selection.py` |
| `tttrlib.Correlator` | `artifact` | `tttr: TTTRRef` | `curve: TableRef` | Multi-tau correlation; mirrors `examples/correlation/plot_normal_correlation.py` |
| `tttrlib.CLSMImage` | `artifact` | `tttr: TTTRRef` | `clsm: ObjectRef` | Construct scanning image from TTTR with scan markers |
| `tttrlib.CLSMImage.compute_ics` | `artifact` | `clsm: ObjectRef` | `ics: BioImageRef`, `summary: TableRef` (if `include_summary=true`) | ICS; mirrors `examples/image_correlation/plot_imaging_ics_tttrlib.py` |
| `tttrlib.TTTR.write` | `artifact` | `tttr: TTTRRef` | `tttr: TTTRRef` | Export to Photon-HDF5 (or other format); uses native `write()` method |

### 8.2 Parameter highlights

Parameters mirror tttrlib's actual API signatures.

- `tttrlib.TTTR` (constructor)
  - `filename: str` (required) — path to TTTR file (must be readable per the server filesystem read allowlist; inherits client permissions)
  - `container_type: str` (optional): `PTU`, `HT3`, `SPC-130`, `SPC-630_256`, `SPC-630_4096`, `HDF` (Photon-HDF5)

- `tttrlib.Correlator` (constructor)
  - `tttr: TTTRRef` (input artifact)
  - `channels: tuple[list[int], list[int]]` — e.g., `([0], [8])` for cross-correlation between ch0 and ch8
  - `n_bins: int` (default 17), `n_casc: int` (default 25), `make_fine: bool` (default False)
  - `method: str` (optional; default `wahl`; one of `wahl`, `felekyan`, `laurence`)
  - Output table columns: `tau` (lag times), `correlation` (G(τ) values)

- `tttrlib.CLSMImage` (constructor)
  - `tttr: TTTRRef` (input artifact)
  - `reading_routine: str` — microscope-specific marker interpretation:
    - `SP5` (Leica SP5)
    - `SP8` (Leica SP8)
    - `default` (generic/PicoQuant)
  - `channels: list[int]`
  - `fill: bool` (default True)

- `tttrlib.CLSMImage.compute_ics`
  - `clsm: ObjectRef` (input artifact — the CLSMImage)
  - `x_range: list[int]`, `y_range: list[int]` (use `[0, -1]` for full frame)
  - `subtract_average: str` (one of `frame`, `stack`, or `""` for no subtraction; default `frame`)
  - `include_summary: bool` (MCP addition; default `false`)

- `tttrlib.TTTR.get_time_window_ranges`
  - `minimum_window_length: float` (seconds)
  - `maximum_window_length: float | None`
  - `minimum_number_of_photons_in_time_window: int`
  - `maximum_number_of_photons_in_time_window: int | None`
  - `invert: bool` (default False)
  - Output table columns: `start_index`, `stop_index`

- `tttrlib.TTTR.header`
  - No additional parameters (serializes `tttr.header.json` to artifact)

- `tttrlib.TTTR.write`
  - `filename: str` (output path; format inferred from extension, e.g., `.h5` for Photon-HDF5)
  - Path rules: relative paths write to the tool run work dir (artifact store); absolute paths are permitted only if allowed by the server filesystem write allowlist (inherits client permissions)
  - Output: `TTTRRef` pointing to the written file

## 9. Smoke Tests Plan (Live Server)

Implementation target: `tests/smoke/test_tttrlib_live.py`.

Tests follow existing style:

- call `list` to verify discovery
- call `describe` on representative `fn_id`s
- call `run` in multi-step workflows
- use `@pytest.mark.requires_env("bioimage-mcp-tttrlib")` and skip if datasets are missing

All paths below refer to the vendored dataset root: `datasets/tttr-data/`.

### 9.1 Smoke: FCS workflow

- Upstream reference: `examples/correlation/plot_normal_correlation.py`
- Dataset: `datasets/tttr-data/bh/bh_spc132.spc`

**MCP call sequence (conceptual)**

1. `list`
2. `describe(fn_id="tttrlib.Correlator")`
3. `run(fn_id="tttrlib.TTTR", inputs={}, params={"filename": <abs>, "container_type": "SPC-130"})` → `tttr` (TTTRRef)
4. `run(fn_id="tttrlib.Correlator", inputs={"tttr": tttr}, params={"channels": [[0], [8]], "n_bins": 7, "n_casc": 27, "make_fine": false})` → `curve` (TableRef)

**Minimal assertions**

- Each step returns an artifact reference (dict with non-empty `ref_id` + `uri`).
- `curve` is a `TableRef` with `row_count > 0`.
- Table columns include `tau` and `correlation`.

### 9.2 Smoke: ICS workflow

- Upstream reference: `examples/image_correlation/plot_imaging_ics_tttrlib.py`
- Primary dataset: `datasets/tttr-data/imaging/leica/sp5/LSM_1.ptu`
- Optional fallback dataset: `datasets/tttr-data/imaging/pq/ht3/pq_ht3_clsm.ht3` (requires `reading_routine: "default"` instead of `"SP5"`)

**MCP call sequence (conceptual)**

1. `describe(fn_id="tttrlib.CLSMImage.compute_ics")`
2. `run(fn_id="tttrlib.TTTR", inputs={}, params={"filename": <abs>, "container_type": "PTU"})` → `tttr` (TTTRRef)
3. `run(fn_id="tttrlib.CLSMImage", inputs={"tttr": tttr}, params={"reading_routine": "SP5", "channels": [0], "fill": true})` → `clsm` (ObjectRef)
4. `run(fn_id="tttrlib.CLSMImage.compute_ics", inputs={"clsm": clsm}, params={"x_range": [0, -1], "y_range": [0, -1], "subtract_average": "frame", "include_summary": false})` → `ics` (BioImageRef)

**Minimal assertions**

- `ics` is a `BioImageRef` and is file-backed (OME-TIFF preferred).
- `ics.metadata.shape` exists and is non-empty.

### 9.3 Smoke: Single-molecule burst selection workflow

- Upstream reference: `examples/single_molecule/plot_single_molecule_burst_selection.py`
- Dataset: `datasets/tttr-data/bh/bh_spc132.spc`

**MCP call sequence (conceptual)**

1. `run(fn_id="tttrlib.TTTR", inputs={}, params={"filename": <abs>, "container_type": "SPC-130"})` → `tttr` (TTTRRef)
2. `run(fn_id="tttrlib.TTTR.get_time_window_ranges", inputs={"tttr": tttr}, params={"minimum_window_length": 0.002, "minimum_number_of_photons_in_time_window": 40})` → `ranges` (TableRef)

**Minimal assertions**

- `ranges` is a `TableRef` with `row_count > 0` and columns include `start_index`, `stop_index`.

### 9.4 Smoke: Photon-HDF5 import/export

- Import dataset: `datasets/tttr-data/hdf/1a_1b_Mix.hdf5`

**MCP call sequence (conceptual)**

1. `describe(fn_id="tttrlib.TTTR.write")`
2. `run(fn_id="tttrlib.TTTR", inputs={}, params={"filename": <abs>, "container_type": "HDF"})` → `tttr_hdf` (TTTRRef)
3. `run(fn_id="tttrlib.TTTR.header", inputs={"tttr": tttr_hdf})` → `header_json` (NativeOutputRef)
4. Export (optional): load `bh_spc132.spc` as in 9.1 and call `run(fn_id="tttrlib.TTTR.write", inputs={"tttr": tttr}, params={"filename": <output.h5>})` → `tttr_exported_hdf` (TTTRRef)
5. Re-load exported `tttr_exported_hdf` via `tttrlib.TTTR` and verify it is readable.

**Minimal assertions**

- `tttr_hdf` is a `TTTRRef` with format `HDF`.
- `header_json` is a `NativeOutputRef` with format `json`.
- Export (if enabled): exported `TTTRRef` is readable via `tttrlib.TTTR`.

## 11. Risks & Mitigations

- **Binary compatibility / Python versions**: mitigate by isolating in a py312 tool env.
- **SWIG / missing signatures**: mitigate by manual schema generation from documentation + schema-driven adaptor.
- **Photon-HDF5 writer availability**: if upstream `tttrlib` does not provide a writer, defer export functionality to a later milestone (do not implement custom h5py writer in v0).
- **Repository bloat from binary datasets**: mitigated by using Git LFS for `datasets/tttr-data/` (see Section 12).
- **Scan-marker variability**: start with the upstream example routine(s) (e.g., `SP5`) and document limitations.

---

## 12. Implementation Decisions (Resolved)

1. **Dataset storage mechanics**: Use **Git LFS** for `datasets/tttr-data/` binaries.
2. **Photon-HDF5 export implementation**: Use upstream `tttrlib` writer if available; if not, defer Photon-HDF5 export to a later milestone.
3. **Schema scraping mechanism**: Parse `tttrlib` repo doc sources (`doc/modules/*.ipynb`, `.rst`) for parameter definitions and examples.
